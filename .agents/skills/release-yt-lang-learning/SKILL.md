# Релиз YT Lang Learning

Выпускай релиз только из корня репозитория `el-yt-lang-learning`. Отвечай пользователю по-русски и не выводи значение `GITHUB_TOKEN` в команды, логи или итоговый ответ.

## Источник истины

1. Полностью прочитай `HOW-TO-RELEASE.md`, `package.json`, `scripts/release.js` и `forge.config.js` перед действиями.
2. Прочитай корневой `AGENTS.md`, если он доступен, и соблюдай его требования.
3. При расхождении этого навыка с актуальным кодом сначала установи фактическое поведение скрипта. Обнови документацию или навык только в рамках запроса пользователя.

## Определи версию

- Используй `patch`, если пользователь не указал тип повышения версии.
- Используй `minor` или `major` только по явному указанию пользователя.
- До запуска вычисли ожидаемую версию из `package.json` и проверь, что соответствующего локального или удалённого тега ещё нет.
- Не изменяй версию вручную перед `bun run release`: release-скрипт делает это сам.

## Подготовь описание релиза

Перед публикацией подготовь Markdown-описание для GitHub Release на украинском языке.

1. Найди две предыдущие опубликованные версии по semver-тегам `vX.Y.Z`. Используй теги, а не дату коммитов. Если тегов меньше двух, возьми самый старый доступный тег.
2. Возьми базой более старую из двух найденных версий и проанализируй изменения от неё до текущего `HEAD`, который будет выпущен:
   - `git log --oneline <baseTag>..HEAD`;
   - `git diff --stat <baseTag>..HEAD`;
   - при необходимости `git diff --name-only <baseTag>..HEAD` и точечный просмотр изменённых пользовательских файлов.
3. Отбери только пользовательски значимые изменения: поведение приложения, UX, импорт/экспорт, субтитры, перевод, плеер, упаковку, обновления, документацию для пользователей. Не включай внутренний шум, если он не объясняет заметное изменение.
4. Пиши кратко и понятно для обычного пользователя. Обычно достаточно 3-6 пунктов. Не перечисляй коммиты, файлы, имена внутренних функций и технические детали, если пользователь не заметит их в приложении.
5. Составь черновик на украинском в формате:

```markdown
## Що змінилося

- ...
- ...

## Перевірка

- Тести пройшли.
- Інсталятор Windows зібрано.
```

6. Обработай черновик по навыку `humanizer-uk`: убери канцелярит, воду, AI-шаблоны, длинное тире и неподтверждённые выводы. Сохрани факты и не добавляй деталей, которых нет в Git.
7. Запиши итоговый Markdown во временный файл вне репозитория, например `$env:TEMP\yt-lang-learning-release-notes-vX.Y.Z.md`. Не создавай release notes в корне проекта, чтобы не загрязнять рабочее дерево перед релизом.
8. Если по Git нельзя уверенно понять пользовательский смысл изменений, остановись и спроси пользователя. Не придумывай пункты описания релиза.

## Предварительная проверка

1. Убедись, что текущая ветка — `main`.
2. Выполни `git fetch origin` и изучи `git status --short --branch`.
3. Не собирай релиз из незафиксированных файлов. Если рабочее дерево грязное, изучи изменения:
   - зафиксируй относящиеся к релизу изменения осмысленным коммитом, если намерение однозначно;
   - не включай посторонние или непонятные изменения без подтверждения пользователя.
4. Убедись, что `main` не отстаёт и не расходится с `origin/main`. Для обычного отставания используй `git pull --ff-only`.
5. Проверь наличие `.env` и строки `GITHUB_TOKEN`, не печатая её значение.
6. Проверь, что `.env` игнорируется Git и исключается в `packagerConfig.ignore` по сегменту пути. Широкий шаблон `/out/` не используй: он способен исключить файлы вроде `player-layout.js`.

Если есть сомнение, какие локальные изменения должны войти в релиз, остановись и спроси пользователя. Установщик и Git должны описывать один и тот же исходный код.

## Проверка пакета

Запусти:

```powershell
bun run test
bun run package
& .\.agents\skills\release-yt-lang-learning\scripts\verify-release-package.ps1
```

Скрипт должен подтвердить:

- наличие `app.asar`;
- наличие `index.html`, `main.js`, `preload.js` и всех локальных скриптов из `index.html`;
- отсутствие `.env` и его потенциально секретных вариантов в архиве. Безопасный шаблон `.env.example` разрешён.

Не продолжай публикацию при ошибке проверки. Сначала получи подтверждённую причину и исправь упаковку. Не меняй несколько неподтверждённых гипотез подряд.

## Публикация

Запусти ровно один раз:

```powershell
bun run release --notes $env:TEMP\yt-lang-learning-release-notes-vX.Y.Z.md
```

Для явно выбранного типа версии передай `patch`, `minor` или `major` перед `--notes`, например:

```powershell
bun run release minor --notes $env:TEMP\yt-lang-learning-release-notes-vX.Y.Z.md
```

Следи за выводом до полного завершения процесса. Скрипт должен:

1. повысить версию;
2. повторно запустить тесты;
3. собрать установщик;
4. создать релизный коммит и тег;
5. отправить ветку и тег;
6. создать GitHub Release с подготовленным украинским описанием;
7. загрузить `yt-lang-learning-setup.exe`.

Не считай задачу законченной, пока процесс ещё выполняется.

## Итоговая проверка

После успеха проверь:

```powershell
git status --short --branch
git rev-parse HEAD
git rev-parse vX.Y.Z
```

Также проверь существование `out/make/squirrel.windows/x64/yt-lang-learning-setup.exe`, GitHub Release по напечатанной ссылке и установщик в Assets. Условия успеха:

- рабочее дерево чистое;
- `main` синхронизирован с `origin/main`;
- тег указывает на `HEAD`;
- релиз опубликован, установщик загружен;
- `.env` и его потенциально секретные варианты отсутствуют в итоговом `app.asar`.

Сообщи пользователю новую версию, ссылку на релиз, результат тестов, имя установщика и состояние Git. Если релиз был создан без подготовленного описания, отдельно скажи, что описание осталось шаблонным и его нужно заполнить вручную.

## Восстановление после ошибки

Не запускай `bun run release` повторно, пока не определишь, на каком шаге он остановился: новый запуск снова повысит версию.

1. Проверь `git status`, `git log -1`, локальные и удалённые теги, страницу GitHub Releases и наличие asset.
2. Если релизный коммит и тег ещё не созданы, а изменён только `package.json`, восстанови его из `HEAD`, устрани причину и затем повтори полный запуск.
3. Если коммит или тег уже создан, не повышай версию снова. Заверши оставшиеся шаги вручную для этой же версии либо, после проверки последствий, удали неполный релиз и тег и повтори процесс.
4. При ошибке Squirrel `Can not access a closed Stream` удали только сгенерированный каталог `out/make/squirrel.windows/x64`. Повтори сборку с `DEBUG=electron-winstaller*`; затем вернись к соответствующему состоянию процесса, не создавая лишний bump версии.
5. Если опубликованный установщик содержал `.env`, немедленно убери с публикации скомпрометированный asset или релиз, удали ошибочный тег при необходимости, исправь упаковку и опубликуй чистый артефакт. Сообщи пользователю, что GitHub-токен нужно отозвать и создать заново.

Перед удалением удалённого релиза или тега ясно установи, что это именно неполный или скомпрометированный релиз текущего запуска. Не затрагивай предыдущие стабильные версии.
